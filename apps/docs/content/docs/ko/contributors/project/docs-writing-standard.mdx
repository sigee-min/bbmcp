# 문서 작성 표준

이 표준은 Ashfox 문서를 사람이 빠르게 작업에 활용할 수 있게 유지하면서, 검색엔진과 답변형 검색에서 문서가 안정적으로 발견되도록 작성/리뷰 기준을 정의합니다.

## 1) 최우선 목표

문서의 1차 KPI는 조회수가 아니라 작업 성공률입니다.

- 독자는 이 페이지만으로 목표 작업을 완료할 수 있어야 합니다.
- 검색 최적화는 이해를 돕는 수단이어야 합니다.
- SEO 기법과 가독성이 충돌하면 가독성을 우선합니다.

## 2) 페이지 범위 규칙

- 한 페이지는 한 사용자 의도에 집중합니다.
- 주제별 canonical URL을 하나로 고정합니다.
- 서로 다른 질문을 동시에 답하려 하면 페이지를 분리합니다.
- 헤딩은 마케팅 문구보다 명확한 명사/동사를 우선합니다.

## 3) 본문과 검색 신호 분리

랭킹 신호는 메타/구조에 두고, 본문은 절차 중심으로 작성합니다.

- 쿼리 친화 표현은 `title`, `description`, 섹션 헤딩에 배치합니다.
- 본문은 실행 절차와 예제를 중심으로 유지합니다.
- 랭킹 목적의 키워드 반복 문장을 넣지 않습니다.
- 구조화 메타데이터는 실제 본문과 일치할 때만 추가합니다.

## 4) 문서 템플릿

문서 유형에 맞는 템플릿을 사용합니다.

### 작업/가이드 문서

1. 목적 (이 문서로 완료할 작업)
2. 사전 준비
3. Quick start (최소 동작 경로)
4. 상세 단계와 옵션
5. 실패 케이스와 복구
6. 관련 링크

### 도구/API 레퍼런스 문서

1. 개요와 적용 범위
2. 입력 스키마
3. 필드 레퍼런스
4. 최소 요청 예시
5. 응답 예시
6. 알려진 제약

## 5) 금지 패턴

아래 항목이 있으면 머지하지 않습니다.

- 키워드 스터핑 또는 부자연스러운 반복 문장
- 실행 정보 없는 일반론/홍보성 문장
- 성능/호환성/정책 관련 미검증 주장
- 크롤러 노출만을 위한 숨김/중복 텍스트
- 유지보수자 검수 없는 대량 AI 생성 텍스트

## 6) PR 리뷰 게이트

문서 PR은 아래 기준을 모두 통과해야 합니다.

- [ ] 초보자도 10분 내 목표 작업을 완료할 수 있다.
- [ ] frontmatter(`title`, `description`)가 유효하다.
- [ ] 내부 링크가 깨지지 않는다.
- [ ] 예제가 실행 가능하거나 스키매틱임을 명시했다.
- [ ] 용어가 기존 문서와 일관된다.
- [ ] 사용자 중심 문장이며 키워드 스터핑이 없다.

## 7) 필수 검증 명령

저장소 루트에서 실행합니다.

```bash
npm run docs:quality
npm --workspace ./apps/docs run build
```

## 8) 로케일 운영 정책

- `en`/`ko` 쌍 문서는 의미적 동등성을 유지합니다.
- 한쪽만 변경되면 짝 로케일 후속 이슈를 생성합니다.
- 명령어/식별자 예시는 로케일과 무관하게 동일하게 유지합니다.
